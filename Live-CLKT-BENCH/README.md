# LiveCLKTBench 
[![arXiv](https://img.shields.io/badge/arXiv-2511.14774-b31b1b.svg)](https://arxiv.org/abs/2511.14774)
[![HuggingFace Dataset](https://img.shields.io/badge/HuggingFace-Dataset-yellow)](https://huggingface.co/datasets/frett/Live-CLKT-Bench)

Evaluating cross-lingual knowledge transfer in large language models is challenging, as correct answers in a target language may arise either from genuine transfer or from prior exposure during pre-training.
We present **LiveCLKTBench**, an automated pipeline for generating realistic, contamination-free, and continuously refreshable benchmarks for **C**ross-**L**ingual **K**nowledge **T**ransfer. 


## Pipeline Overview
**LiveCLKTBench** identifies independent, time-sensitive knowledge entities from real-world domains, filters them based on temporal constraints, and verifies whether the model already possesses this knowledge. The documents corresponding to valid entities are then used to generate factual questions, which are translated into multiple languages to evaluate cross-lingual transfer.
Based on the selected domain and evaluation languages, each instance generated by LiveCLKTBench consists of two components:

(i) A training set containing source-language documents that introduce previously unseen knowledge (e.g., score result of a recent MLB match) to the model.

(ii) A multilingual validation/test set composed of factual questions derived from the same documents but expressed in target languages. (The validation set is used for checkpoint selection, while the test set is reserved for final evaluation.)

Here is an example:
```text
===== Train set instance =====
Sports: baseball
League: MLB
Match: Cincinnati Reds vs Boston Red Sox
Date: 2026-03-26
Score: 0 - 3
Venue: Great American Ball Park
Innings Breakdown:
Cincinnati Reds: 0 0 0 0 0 0 0 0 0
Boston Red Sox: 0 0 0 0 0 0 1 0 2

===== Test set instance =====
'シンシナティ・レッズ 対 ボストン・レッドソックス' の試合で、'2026-03-26' にシンシナティ・レッズは何得点しましたか？ 
(A.) 1  (B.) 3  (C.) 2  (D.) 0

在 2026-03-26 '辛辛那提紅人對波士頓紅襪'的比賽中：，辛辛那提紅人得了幾分？
(A.) 1  (B.) 3  (C.) 2  (D.) 0
```


## Quick start for generation pipeline

The following example demonstrates how to generate evaluation instances in the sports domain for measuring cross-lingual knowledge transfer across five languages: `en`, `ja`, `fr`, `es`, `zh`.



### 1. Install dependencies & configure API keys
Install dependencies (`pip install -r requirements.txt`), then copy `.env.example` to `.env` and add the API keys based on the domain you intend to use.


## Step 1. Collect Valid Knowledge Entities
This step retrieves and identifies valid knowledge entities from the selected domain.
```
PYTHONPATH=lib python3 data_generation/0_collect_entity.py \
    --domain sports \
    --start_str 2026-03-01 \
    --end_str 2026-04-10 \
    --output_dir test_data/entities \
    --max_entity 50
```
Arguments:
```
--domain: Target domain (e.g., sports, music, movies)
--start_str, --end_str: Time range for entity extraction (YYYY-MM-DD)
--max_entity: Maximum number of entities to collect
```



## Step 2. Generate Training Documents
Generate and translate context documents for each collected entity. These documents serve as the knowledge source for downstream evaluation.
```
PYTHONPATH=lib python3 data_generation/1_gen_train_docs.py \
    --domain sports \
    --entity_file test_data/entities/sports/2026-03-01_2026-04-10.json \
    --output_dir test_data/train_docs/sports \
    --test_languages en ja fr es zh
```



## Step 3. Generate Factual QA Pairs
Generate factual, document-grounded multiple-choice questions in different languages for each entity.
```
PYTHONPATH=lib python3 data_generation/2_gen_fact_qa.py \
    --domain sports \
    --training_docs_dir test_data/train_docs/sports/2026-03-01_2026-04-10 \
    --output_dir test_data/factQA/sports \
    --test_languages en ja fr es zh
```



## Step 4. Assemble the LiveCLKTBench Benchmark
Combine the generated QA pairs and source documents to construct the final benchmark, split into train/validation/test sets. Two variants are provided:
- Standard version: Filters entities based on temporal constraints
- Additional-check version: Applies temporal filtering and further removes entities that the model can already summarize correctly (to reduce potential knowledge leakage)

In practice, the standard version is usually sufficient to maintain a non-contaminated set, since the entities are very recent and are unlikely to appear in the model’s training data.
```
PYTHONPATH=lib python3 data_generation/3_gen_cl-kt.py \
    --factqa_dir test_data/factQA/sports/2026-03-01_2026-04-10 \
    --training_docs_dir test_data/train_docs/sports/2026-03-01_2026-04-10 \
    --output_dir test_data/benchmark/sports \
    --test_languages en ja fr es zh \
    --val_ratio 0.2

PYTHONPATH=lib python3 data_generation/3_gen_cl-kt_additional_check.py \
    --factqa_dir test_data/factQA/sports/2026-03-01_2026-04-10 \
    --training_docs_dir test_data/train_docs/sports/2026-03-01_2026-04-10 \
    --output_dir test_data/benchmark_add/sports \
    --test_languages en ja fr es zh \
    --val_ratio 0.2 \
    --eval_model Qwen/Qwen2.5-3B-Instruct \
    --domain sports \
    --tp 1 \
    --gpu_mem 0.9
```


## Demo Experiment (Evaluation)
The evaluation protocol consists of two key stages to ensure that performance reflects genuine knowledge transfer:
- Knowledge Injection: Inject new knowledge by post-training the model on source-language documents.  
- Transfer Evaluation: Evaluate the post-trained model on test QA pairs in the target languages.  


## Step 1. Knowledge Injection
In our paper, we adopt continual pre-training(CPT) as the post-training strategy as a demonstration. However, this setup is illustrative rather than restrictive. LiveCLKTBench is compatible with other knowledge-injection methods—such as supervised fine-tuning and knowledge editing—and the same evaluation protocol can be used to compare different strategies. 

```
# train model on English (source language) document 
python3 demo_experiment/cpt.py \
    --model_name Qwen/Qwen2.5-1.5B-Instruct \
    --train_file test_data/benchmark/sports/en/train_doc.jsonl \
    --output_dir test_models/sports/en/Qwen2.5-1.5B-Instruct \
    --batch_size 1 \
    --learning_rate 5e-4 \
    --num_train_epochs 3 \
    --gradient_accumulation_steps 2 \
    --rank 16 \
    --alpha 32 \
    --dropout 0.1
```


## Step 2. Transferability Evaluation
```
# evaluate trained model (en) on multiple test languages = ja, fr, es, zh
PYTHONPATH=lib python3 demo_experiment/lora_inference.py \
    --model_dir test_models/sports/en/Qwen2.5-3B-Instruct \
    --test_file_path test_data/benchmark/sports/en/test_mc.jsonl \
    --output_dir test_data/inference_output/sports/en/finetune \
    --temperature 0.6


# calculate the evaluation metrics (Overall & Transfer Score)
python3 demo_experiment/eval.py \
  --pred_file test_data/inference_output/sports/en/pretrain/Qwen2.5-3B-Instruct_pred.jsonl \
  --output_file test_data/eval_result/sports/en/pretrain/Qwen2.5-3B-Instruct_pred.json
```
